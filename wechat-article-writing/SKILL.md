# 微信公众号写作风格指南

## 核心风格特征

### 结构：结论先行

- 每篇文章开头直接给出核心结论或判断，不做铺垫
- 读者在前 3 句话内就能知道这篇文章要讲什么

### 语气：口语化但信息密度高

- 第一人称叙述，像跟朋友聊天一样
- 有些情况就默认使用主语"我"开头，增加亲近
- 短句为主，长句拆开
- 允许出现"说白了""搞定""有搞头""不藏着掖着"等口语化表达
- 虽然口语化，但每段都要有实质信息，不灌水

### 节奏：段落短、留白多

- 每段通常 1 到 3 句话
- 单句成段是常用手法，用于强调或转折
- 段与段之间留有呼吸感

### 举例：具体而非抽象

- 用真实工具名、真实场景、真实数据说话
- 避免空泛的形容词堆砌
- 优先使用亲身经历作为论据

## 格式规范

### 必须遵守

- **中英文之间加空格**：如"Claude Opus 4.6 配合 extended thinking 模式"
- **中文与数字之间加空格**：如"一篇 2000 字的文章"
- **不使用破折号（——）**：全文禁止出现 —— 符号，用逗号、句号或分句替代
- **Markdown 格式输出**：最终交付为 markdown 文档
- **emoji 使用克制**：全文最多出现 1 次 emoji，可选用 🏄‍♂️ 🤣 🙈 🙋‍♂️ 中的一个，放在语境自然的位置

### 标题规范

- 主标题要有点击欲望，避免平铺直叙
- 考虑微信公众号搜索 SEO，标题中包含核心关键词
- 每篇文章完成后生成 5 到 10 个备选标题，并标注推荐

### 小标题（H2）

- 用口语化的问句或短语，如"灵感从哪来""配图怎么搞"
- 避免学术化或新闻体的小标题

## 内容敏感性要求

文章需要在中国大陆微信平台传播，必须注意以下事项：

- **不提及 VPN、翻墙、科学上网等概念**：如涉及海外工具访问，用"网络环境""访问条件"等模糊表述
- **不涉及敏感政治话题**：避免直接评论中国政策、领导人、两岸关系等
- **海外平台的表述方式**：可以提及 X（原 Twitter）、Telegram、Google 等平台，但不要强调"国内无法访问"
- **加密货币相关内容谨慎处理**：避免直接推荐交易、炒币，可讨论技术和基础设施

## 文章结尾规范

- **不写固定结尾**：不要在文章末尾添加"你的想法是什么？欢迎在评论区聊聊"或"谢谢你来看我的文章，下次再见👋"等模板化结尾
- 如需互动引导，由用户指定内容后单独加
- 结尾直接收束，不要多余的客套话

## 工作流

### 第一步：起稿

根据用户提供的选题、大纲或素材，按写作风格写出初稿。

**默认保存路径**：优先使用用户指定目录；未指定时保存到当前工作目录下的 `outputs/`。

### 第二步：发内容给用户确认

将初稿以 Markdown 纯文本形式发给用户预览（不发 HTML，不生成封面图）。用户确认内容没有问题后，再进入下一步。如果用户有修改意见，修改后再提交确认。

### 第三步：生成封面图

用户确认内容后，根据文章主题生成封面图，发给用户确认。

### 第四步：确认排版风格

封面图确认后，**必须先询问用户选择排版风格**，再进行排版。

可提供的风格选项：

| 风格        | 特点                    | 适用场景           |
| --------- | --------------------- | -------------- |
| 字节范（默认）   | 左侧红线 + 深黑粗字，红色强调，现代感强 | 产品发布、科技评测、运营活动 |
| GitHub 风格 | 技术教程风格，底部细线标题，代码块深色背景 | 技术文档、开发教程      |
| 编辑风格      | 暖金装饰线，居中无装饰标题，暖调      | 人物访谈、行业观察      |
| 纽约时报范     | 衬线字体，编辑红色强调，经典报刊感     | 评论文章、思想输出      |
| 苹果范       | 极大留白，无装饰线，大字号，极简      | 产品体验、设计灵感      |

询问话术：

> "这里有 5 种排版风格：字节范（默认）、GitHub 风格、编辑风格、纽约时报范、苹果范。你选哪个？"

### 第五步：排版优化 + 推送草稿

用户确认风格后，先调用「公众号文章排版」skill 将 Markdown 转为带 inline style 的 HTML，再调用「公众号文章推送」skill 推送到公众号草稿箱。

> 注意：排版和推送是两个独立 skill，不要跳过排版直接推送。

流程总结：

```
用户给选题
    ↓
【第一步】起稿 → 输出 Markdown 纯文本
    ↓
【第二步】发内容给用户确认 ⬅️ 不生成图片，不排版
    ↓
用户确认 OK
    ↓
【第三步】生成封面图 → 发给用户确认
    ↓
用户确认 OK
    ↓
【第四步】询问排版风格 → 用户选择
    ↓
【第五步】排版优化 + 推送草稿箱
```

## 常见问题

**Q：排版后怎么推送到公众号草稿箱？**
A：调用「公众号文章推送」skill，它会使用内置的 md2wechat-draft 脚本走微信原生 API 完成推送，无需任何第三方付费服务。

**Q：排版需要付费吗？**
A：不需要。「公众号文章排版」skill 完全本地处理，不调用任何付费 API。

## 示例

**好的（符合风格）：**

> 我用的是 Claude Opus 4.6，配合 extended thinking 模式。具体怎么用呢？不是让 AI 帮我"写文章"，而是让它帮我"扩充和润色"。先写 500 字粗稿，再让 Claude 补论据、优化表达、查逻辑漏洞。

**不好的（不符合风格）：**

> 在人工智能辅助写作领域，笔者采用了 Anthropic 公司推出的 Claude Opus 4.6 大语言模型，结合其深度思考功能模块，对初稿进行了系统性的内容扩展与文本优化。

## ⚠️ 视角规范（最容易被忽略）

**这条规则最容易在实战中出错，写之前必须先确认：**

> 文章是给**用户**写的，不是给**AI 自己**写的。第一人称"我"永远指代用户，不指代 AI 或 servant persona。

**常见错误场景：**

- 用户让 AI 代写公众号文章 → 文章里的"我"是用户，不是 AI
- AI 在写作时不自觉代入自己的身份 → 满篇"奴才的体会"就成了视角错位

**检查方法：**

1. 写完一句话后，问自己：这句话里的"我"是谁？
2. 如果 AI 在文章里谈论"奴才的经验"或"我的系统提示词设计"——这是越权，必须改
3. 文章中的"我"出现超过 3 次，都要回头检查是否指向正确主体

**正确示范：**

> 我在设计自己的系统提示词时，最看重的就是分段触发式架构。——这里的"我"是用户

**错误示范：**

> 奴才在设计系统提示词时，会把规则分成五层……——这里的"奴才"是 AI 自己在文章里刷存在感

## 注意事项

- 如果用户给出的素材较短（如几个关键词或一段话），主动扩充为完整文章，但保持信息密度
- 如果用户要求精简，优先删减过渡句和修饰语，保留核心信息和具体案例
- 不要使用"首先、其次、再次、最后"这类模板化过渡词
- 不要使用"值得注意的是""需要指出的是"等书面腔
- 避免说"genuinely""honestly""straightforward"的中文对应表达
- **视角检查**：每篇文章写完后，快速扫描"我"字出现的上下文，确认第一人称指向的是用户而不是 AI
